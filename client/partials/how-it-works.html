This diagram, courtesy of MDN's WebRTC connectivity article, gives a sense of everything that Icecomm abstracts away for developers. Essentially, in order for a peer-to-peer connection to happen between browsers, a framwork known as Interactive Connectivity Establishment (ICE) must be used to exchange the information neccessary for the browsers to connect directly. As you may have surmised, the name 'Icecomm' was derived from this framework- with the 'comm' being shorthand for 'communication.'

With WebRTC, the ICE candidates exchange information via STUN (Session Traversal Utilities for NAT [Network Address Translation]) and TURN (Traversal Using Relays around NAT) servers. SDP (Session Description Protocol) information is passed between peers, which translates the data into a standard readable by all connected peers. For developers using Icecomm, this server communication is all handled behind the scenes.
